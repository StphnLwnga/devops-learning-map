Networking Fundamentals

Tier 1 · Foundations

SSH, DNS, certificates, HTTP/HTTPS, SSL/TLS, mTLS, and APIs — the networking and security foundations every DevOps engineer needs.

SSH & SSH Keys

SSH (Secure Shell) is a cryptographic network protocol used to securely connect to remote servers. It encrypts all communication between your machine and the server, preventing eavesdropping and tampering. SSH operates on port 22 by default and is formalized in IETF standards (RFC 4251–4254).

The SSH protocol is built on three layers:

  • Transport Layer — Handles symmetric encryptionA type of encryption where the same key is used to both encrypt and decrypt data. Fast and efficient for bulk data transfer., key exchange, and server authentication. Ensures data integrity.
  • Authentication Layer — Verifies the user’s identity via passwords, public keys, or other methods.
  • Connection Layer — Multiplexes the encrypted tunnel into logical channels for shell sessions, port forwarding, and SFTPSSH File Transfer Protocol — a secure file transfer protocol that runs over an SSH connection. Replaces insecure FTP..

What is an SSH Key?

An SSH key is a way to authenticate without passwords. Instead of typing a password each time, you use asymmetric encryptionA cryptographic system using two mathematically linked keys: a public key (shared freely) and a private key (kept secret). Data encrypted with one can only be decrypted by the other. — a mathematically related public key and private key. This approach is more secure than passwords because the private key never travels over the network.

Two Parts of an SSH Key

  • Public Key — Shared with servers you want to access. Think of it like a padlock you can distribute freely.
  • Private Key — Kept secret on your local machine. This is the only key that can “unlock” data encrypted with the matching public key.
⚠️
Never share your private key. Only the public key (id_rsa.pub) goes on remote servers. If your private key is compromised, regenerate your key pair immediately.

How SSH Keys Work

1

Client contacts server — Your machine requests an SSH connection to the remote server.

2

Server sends its public key — The server identifies itself so you can verify you’re connecting to the right host.

3

Parameters negotiated — Both sides agree on encryption algorithms and establish a secure, encrypted channel.

4

Client authenticates — You prove your identity using your private key. No password is transmitted over the network.

SSH Key Algorithms

Algorithm Key Size Notes
RSA 2048–4096 bits Established and widely supported. Use 4096 bits minimum for new keys.
ECDSA 256, 384, or 521 bits Elliptic curve standard. 521 bits preferred. Smaller keys than RSA for equivalent strength.
Ed25519 Fixed (256-bit curve) Modern and fastest. Recommended for new setups where client support is available.

Generating SSH Keys

Use the ssh-keygen command to create a new key pair:

bash 
# Generate a 4096-bit RSA key
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"

# Generate a modern Ed25519 key (recommended)
ssh-keygen -t ed25519 -C "your_email@example.com"

# Generate a key with a custom filename
ssh-keygen -t ed25519 -f ~/.ssh/my_project_key

Key File Locations

SSH keys are stored in the ~/.ssh directory:

  • ~/.ssh/id_rsa — Your private key (never share this)
  • ~/.ssh/id_rsa.pub — Your public key (copy this to servers)
  • ~/.ssh/authorized_keys — On the server, lists public keys allowed to connect
  • ~/.ssh/known_hosts — Records fingerprints of servers you’ve connected to before

Why Use SSH Keys?

  • More secure — Keys are much harder to brute-force than passwords
  • No password needed — Enables automated, unattended connections
  • Essential for DevOps — Used in CI/CD pipelines, server management, Git authentication, and automation

DNS & Certificates

What is a Certificate?

A digital certificate is an electronic document that proves the identity of a server. It’s used in HTTPS to ensure you’re communicating with the legitimate server and not an imposter. Certificates are issued by trusted Certificate Authorities (CAs)A trusted organization that issues digital certificates. Examples include Let’s Encrypt, DigiCert, and Comodo. CAs verify the identity of certificate applicants before issuing..

Types of Certificates

Root Certificate

The top-level trusted certificate. Stored in your OS and browser. Issued by a Certificate Authority like DigiCert or Let’s Encrypt.

Intermediate Certificate

A bridge between the root and server certificate. Improves security by keeping the root CA offline. Signed by the root CA.

Server Certificate

Issued to a specific domain (e.g., example.com). Installed on the web server. This is the certificate used for HTTPS connections.

Certificate Chain of Trust

When your browser validates a certificate, it follows the chain from server to root:

Server Certificate
→
Intermediate CA
→
Root CA

The browser validates the entire chain. If any link is broken, untrusted, or expired, the connection is rejected.

What is DNS?

DNS (Domain Name System) is a hierarchical, decentralized naming system that translates human-readable domain names into IP addresses. Without DNS, you’d have to memorize addresses like 142.250.x.x to visit Google.

How DNS Resolution Works

When you type a URL into your browser, a multi-step lookup finds the server’s IP address:

1

Browser cache — Checks if the domain was recently resolved and is still within its TTLTime To Live — the duration (in seconds) a DNS record is cached before the resolver must query again. Low TTL = fresher data but more DNS traffic. Typical values: 300s–86400s..

2

OS cache — The operating system checks its own DNS cache and the /etc/hosts file.

3

Recursive resolverA DNS server (usually run by your ISP or services like 8.8.8.8 / 1.1.1.1) that does the heavy lifting of walking the DNS hierarchy on your behalf. — Your ISP’s DNS server (or a public resolver like 1.1.1.1) begins querying the hierarchy.

4

Root nameserver — Directs the resolver to the correct TLDTop-Level Domain — the last segment of a domain name (.com, .org, .io, .dev). Each TLD has its own set of authoritative nameservers. server (e.g., the .com servers).

5

TLD nameserver — Returns the authoritative nameserver for the specific domain.

6

Authoritative nameserver — Returns the final IP address. The result is cached at each level according to the record’s TTL.

Common DNS Record Types

Record Purpose Example
A Maps domain to an IPv4 address example.com → 192.168.1.1
AAAA Maps domain to an IPv6 address example.com → 2001:0db8:85a3::8a2e:0370:7334
CNAME Maps domain to another domain (alias) www.example.com → example.com
MX Specifies mail server for email routing example.com → mail.example.com
TXT Stores text information for verification SPF, DKIM, domain ownership (Google, AWS)

How DNS + Certificates Work Together

DNS and certificates are complementary. Here’s what happens when you visit https://example.com:

1

DNS resolves the IP — Your browser queries DNS to find the server’s IP address.

2

Server presents its certificate — The server sends its TLS certificate to prove its identity.

3

Browser validates the chain — The certificate chain is checked against trusted root CAs.

4

Secure connection established — An encrypted HTTPS session begins.

💡
Best practices: Keep DNS records updated, use the correct record type for each purpose, and monitor certificate expiration dates to avoid outages.

HTTP vs HTTPS

What is HTTP?

HTTP (HyperText Transfer Protocol) is the foundation of data communication on the web. It’s a statelessEach HTTP request is independent — the server does not remember previous requests. State is maintained through cookies, tokens, or session IDs sent with each request. client-server protocol that defines how messages are formatted and transmitted. However, HTTP sends data in plain text, meaning anyone intercepting the traffic can read it. HTTP uses port 80.

HTTP Request & Response

Every HTTP interaction follows a request/response cycle:

http request 
GET /api/users HTTP/1.1
Host: example.com
Accept: application/json
http response 
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 42

{"id": 1, "name": "Alice"}

What is HTTPS?

HTTPS (HTTP Secure) is HTTP with encryption. It uses SSL/TLS protocols to encrypt all data in transit, ensuring confidentiality, integrity, and authentication. HTTPS uses port 443 and requires a valid TLS certificate.

How HTTPS Works

1

User connects to site — The browser initiates a connection to the server on port 443.

2

Server sends certificate — The server presents its TLS certificate containing its public key.

3

Browser verifies — The browser checks the certificate against trusted CAs and validates the domain.

4

Encrypted connection starts — A symmetric session key is negotiated and all subsequent data is encrypted.

HTTP vs HTTPS Comparison

Feature HTTP HTTPS
Encryption None — plain text SSL/TLS encryption
Port 80 443
URL prefix http:// https://
Certificate Not required TLS certificate required
Data safety Vulnerable to interception Encrypted in transit

Why HTTPS Matters

  • Protects passwords & data — Encrypts sensitive information like login credentials, payment details, and personal data
  • Prevents attacks — Stops man-in-the-middle attacks where attackers intercept or modify traffic
  • Required for modern apps — Browsers flag HTTP sites as “Not Secure,” many APIs require HTTPS, and it’s an SEO ranking factor

SSL vs TLS

What is a Protocol?

A protocol is a set of rules that tells systems how to communicate with each other. It defines how data is sent, received, and secured. Common examples include HTTP (websites), HTTPS (secure websites), SSH (remote server access), and FTP (file transfers).

What is SSL?

SSL (Secure Sockets Layer) was the original protocol for encrypting internet communications. Developed by Netscape in the mid-1990s, SSL is now deprecated due to known security vulnerabilities and weak encryption. It is no longer recommended for use.

What is TLS?

TLS (Transport Layer Security) is the modern, secure replacement for SSL. It provides stronger encryption, better performance, and is the protocol actually used behind HTTPS today. The current version is TLS 1.3.

ℹ️
When people say “SSL” today, they almost always mean TLS. The term “SSL certificate” persists in common usage even though the SSL protocol itself has been deprecated since 2015. The current standards are defined in RFC 5246 (TLS 1.2) and RFC 8446 (TLS 1.3).
⚠️
Browser enforcement: All major browsers dropped support for TLS 1.0 and 1.1 in 2020 (Firefox 74+, Chrome 84+, Safari 13.1+). Servers must support TLS 1.2 or 1.3 to remain accessible.

SSL vs TLS Comparison

Aspect SSL TLS
Status Deprecated Active standard
Security Known vulnerabilities Stronger encryption
Performance Slower handshake Faster (TLS 1.3: 1-RTT)
Used in Legacy systems only HTTPS, email, VPNs

How TLS Works — The Handshake

1

Client Hello — Sends supported TLS versions and cipher suitesA set of cryptographic algorithms negotiated during the TLS handshake. Specifies which algorithms to use for key exchange, bulk encryption, and message authentication. to the server.

2

Server Hello — Chooses a cipher suite and sends its certificate.

3

Client verifies — Validates the server’s certificate against trusted CAs.

4

Keys exchanged — Both sides generate symmetric session keys using forward secrecyA property ensuring that session keys cannot be compromised even if the server’s long-term private key is later exposed. Each session uses unique ephemeral keys..

5

Secure session created — All subsequent communication is encrypted with the session keys.

TLS 1.2 vs TLS 1.3

FeatureTLS 1.2TLS 1.3
Handshake 2 round-trips (RTT)Round-Trip Time — the time it takes for a message to travel from client to server and back. Fewer RTTs mean faster connection setup. 1 RTT (with 0-RTT resumption for returning clients)
Cipher suites Supports legacy ciphers (some weak) Removed all insecure ciphers; only strong defaults
Forward secrecy Optional Mandatory for all connections
Adoption Widely deployed since 2008 Standard since 2018; rapidly adopted

mTLS (Mutual TLS)

Mutual TLS (mTLS) is a security protocol where both the client and server authenticate each other using certificates. In standard TLS, only the server proves its identity. In mTLS, both sides must present and verify certificates before communication can begin.

How mTLS Works

1

Client connects to server

2

Server sends its certificate — Just like in standard TLS.

3

Client verifies the server — Checks the server’s certificate against trusted CAs.

4

Client sends its certificate — This is the “mutual” part — the client also proves its identity.

5

Server verifies the client — Checks the client’s certificate against its trusted CA list.

6

Secure connection established — Both sides are mutually authenticated; encrypted communication begins.

Where is mTLS Used?

Microservices

Service-to-service communication in Kubernetes clusters, managed by service meshesInfrastructure layers (e.g., Istio, Linkerd) that inject sidecar proxies alongside each service. These proxies handle mTLS transparently — applications don’t manage certificates themselves. like Istio or Linkerd.

Internal APIs

B2B integrations and internal API gateways where both parties must verify each other’s identity.

Banking Systems

High-security financial systems where regulatory compliance demands strong mutual authentication.

Zero Trust Security

Zero trust architectures where no connection is trusted by default — every request must be authenticated.

TLS vs mTLS Comparison

Feature TLS mTLS
Encryption Yes Yes
Server authenticated Yes Yes
Client authenticated No (uses passwords/tokens) Yes (via certificate)
Security level Medium High
Complexity Simple More complex (requires client cert management)
ℹ️
When to use each: Use TLS for public websites, basic API security, and login/password systems. Use mTLS for microservices, internal APIs, banking/high-security systems, and zero trustA security model that assumes no user, device, or network is inherently trusted. Every request must be verified, authenticated, and authorized — even from inside the corporate network. architectures.
⚠️
Certificate management overhead: mTLS requires infrastructure to issue, distribute, rotate, and revoke client certificates. Large organizations can accumulate millions of certificates — a Certificate Authority (CA)In mTLS contexts, often an internal/private CA (not a public CA like Let’s Encrypt). Tools like HashiCorp Vault, cert-manager, or SPIFFE/SPIRE automate certificate lifecycle management. and automation tooling are essential.

APIs (Application Programming Interfaces)

An API (Application Programming Interface) is a bridge that allows two applications to communicate with each other. When a mobile app fetches weather data, when a website processes a payment, or when a CI/CD pipeline triggers a deployment — APIs make it happen.

How APIs Work

Client sends request
→
API processes
→
Server responds with data

Types of APIs

REST API

The most common type. Uses standard HTTP methods, statelessEach request contains all the information needed to process it. The server does not store client state between requests., returns JSON. Key constraints: client-server separation, cacheability, and uniform interface.

SOAP API

XML-based protocol with strict contracts (WSDL). Used in enterprise and legacy systems requiring formal specifications.

GraphQL API

Query language where clients request exactly the data they need. Reduces over-fetching and under-fetching.

WebSocket API

Full-duplex, real-time communication. Used for chat apps, live dashboards, and streaming data.

HTTP Methods

Method Purpose Example
GET Retrieve data GET /users → list all users
POST Create new data POST /users → create a user
PUT Update existing data PUT /users/1 → update user 1
DELETE Remove data DELETE /users/1 → delete user 1

API Security

  • Use HTTPS — Always encrypt API traffic to prevent interception
  • Authentication — Verify identity using API keys, OAuth 2.0 tokens, or JWTJSON Web Token — a compact, URL-safe token format containing encoded claims (user ID, roles, expiration). Signed by the server to prevent tampering.
  • Rate limitingRestricting the number of API requests a client can make within a time window (e.g., 100 requests/minute). Prevents abuse, brute-force attacks, and protects server resources. — Prevent abuse by limiting the number of requests per client
  • mTLS for internal APIs — Use mutual TLS for service-to-service communication in microservice architectures
  • Input validation — Sanitize and validate all incoming data to prevent injection attacks

Where APIs Are Used

  • Web apps — Frontend communicating with backend services
  • Mobile apps — Fetching data from remote servers
  • Microservices — Services communicating with each other
  • Cloud services — Managing infrastructure programmatically (AWS, GCP, Azure APIs)
