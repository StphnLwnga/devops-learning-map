Section 01

What is DevOps?

DEV + OPS = DevOps

DevOps is a combination of Development (DEV) and Operations (OPS). It is a culture and set of practices that bridges the gap between Development and Operations teams to improve and better control the software development life cycle (SDLC).

DevOps engineers sit in the middle of all SDLC teams and provide tools, automation, and processes that enable those teams to deliver software faster, more reliably, and with greater collaboration.

Section 02

What is DEV?

DEV stands for Application Development. An application is a program, or set of programs, that allows end-users to perform particular functions.

In simpler terms, applications are the tools we use every day:

  • Facebook and WhatsApp for communication and social media
  • Banking apps for managing finances
  • DoorDash for ordering food
  • Gmail, Yahoo, and Outlook for email

Important distinction: DevOps does not develop applications — developers do that. DevOps engineers are not developers. They work on the operations side of the software development life cycle.

Section 03

What is OPS?

OPS stands for Operations. Operations means transforming the code coming from developers into a fully working application that end-users can interact with.

DevOps engineers use a wide variety of tools, automation pipelines, and infrastructure management practices to accomplish this transformation. This includes building, testing, deploying, monitoring, and maintaining the application in production.

Think of it this way: Developers write the code. Operations teams transform that code into a working, running, accessible application.

Section 05

The DevOps Process

The DevOps process is a continuous loop of activities that work together to deliver software rapidly and reliably. Each phase feeds into the next, creating an infinite feedback loop.

01

Continuous Planning

Continuously defining goals, tasks, and requirements throughout the project lifecycle. Teams collaborate to break down features into actionable work items.

Example: Team meets to discuss adding a discount feature to an e-commerce site, defining user stories and acceptance criteria.
02

Continuous Development

Constantly writing, testing, and building code in small increments. Developers work in short cycles, committing code frequently to version control.

Example: Developers code the input field and discount calculation, testing each part individually before committing.
03

Continuous Integration (CI)

Frequently merging code changes into a shared repository with automated testing. Every integration triggers an automated build and test sequence to catch issues early.

Example: Developer integrates the discount feature into the main codebase; automated tests check compatibility with existing features.
04

Continuous Code Inspection

Reviewing code manually or with automated tools for bugs, security vulnerabilities, and overall code quality. Ensures adherence to coding standards and best practices.

Example: A team member spots a security issue in the discount code validation during a code review, preventing a potential exploit.
05

Continuous Testing

Running automated tests at every stage to validate functionality, performance, and reliability. Includes unit tests, integration tests, end-to-end tests, and performance tests.

Example: Automated test suites run against the discount feature, verifying edge cases like expired codes, max usage limits, and concurrent usage.
06

Continuous Deployment

Automatically deploying code to production after all tests pass. Every change that passes the pipeline reaches production without manual intervention.

Example: The discount feature is automatically deployed to the live website after all automated tests pass successfully.
07

Continuous Delivery (CD)

Ensuring code is always in a deployable state, with a manual approval gate before production releases. Unlike Continuous Deployment, CD requires human sign-off for production pushes.

Example: The discount feature is ready for production but waits for the product manager's approval before going live.
08

Continuous Security

Integrating security practices throughout the entire pipeline. Security is not a phase but a constant practice — often called “shifting security left.”

Example: Security scans automatically run on every commit, checking for dependency vulnerabilities and code-level security flaws.
09

Continuous Feedback

Collecting feedback from users, stakeholders, and monitoring tools to inform development priorities and improvements.

Example: Team monitors user feedback on the discount feature, noticing users want the ability to stack multiple discount codes.
10

Continuous Improvement

Using feedback and retrospective data to make enhancements. Teams regularly reflect on processes, tools, and outcomes to optimize their workflow.

Example: Based on user feedback, the team adds clearer instructions to the discount code field and improves error messages.
11

Continuous Monitoring & Logging

Tracking application performance and recording events in real time. Provides visibility into system health, user behavior, and potential issues before they escalate.

Example: Monitoring tools track discount code usage rates and log errors when invalid codes are submitted, alerting the on-call engineer.

Section 07

DORA Metrics

The DORA (DevOps Research and Assessment) metrics are four key measures that indicate the performance of a software delivery team. They are used industry-wide to benchmark and improve DevOps practices.

🚀

Deployment Frequency

How often your organization successfully releases to production. Elite teams deploy on-demand, multiple times per day.

⏱

Lead Time for Changes

The time it takes for a code change to go from commit to successfully running in production. Shorter is better.

⚠

Change Failure Rate

The percentage of deployments that cause a failure in production requiring remediation (rollback, hotfix, patch).

🔄

Mean Time to Recovery

The average time it takes to recover from a failure in production. Also known as MTTR. Lower is better.

Section 09

PreProd vs Production

Understanding the differences between Pre-Production and Production environments is critical for DevOps engineers.

Aspect Pre-Production Production
Purpose Validate changes in a production-like environment before going live Serve real end-users with the live application
Configuration & Data Uses sanitized or synthetic test data; mirrors prod config Uses real user data and production configurations
Testing & Validation Final round of integration, performance, and UAT testing Monitoring and observability replace active testing
User Access Restricted to internal teams and stakeholders Open to all end-users and the public
Change Control Changes deployed frequently for validation Strict change control with approval processes
Failure & Rollback Failures are expected and used for learning; easy rollback Failures impact users; requires rapid incident response and rollback
Section 10

Deployment vs Release

While often used interchangeably, deployment and release are two distinct concepts in the software delivery process.

Technical

Deployment

Moving code from development/testing environments to production. It is a technical process focused on installing and configuring the application on servers.

Deployments are often internal and may not be immediately visible to end-users. Feature flags or canary releases can hide deployed code from users.

Business

Release

Making a deployed version accessible and visible to end-users. It is a business-oriented process that involves communication with stakeholders, marketing, and support teams.

Releases are external — they represent the moment users can interact with new features.

Key Difference: Deployment is internal and technical — code is placed on servers. Release is external and visible — features are made available to users. You can deploy without releasing (dark launches), and you can release previously deployed code (feature flags).

Section 11

Environment Software Lifecycle

Software versions move through each environment sequentially. A version must successfully pass through all lower environments before being promoted to production.

Version promotion flow

v1.0.0 → v1.0.0 → v1.0.0 → v1.0.0
v2.0.0 → v2.0.0 → v2.0.0 → v2.0.0
v3.0.0 → v3.0.0 → v3.0.0 → v3.0.0
vN.0.0 → vN.0.0 → vN.0.0 → vN.0.0

Each row represents a full cycle: development, quality assurance, pre-production validation, and finally, production deployment. The cycle repeats with every new version, ensuring continuous delivery of improvements.
