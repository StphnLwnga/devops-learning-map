Git & Version Control

Tier 1 · Foundations

Branching, merging, workflows — the backbone of modern software development.

What is Git?

Git is a DevOps tool used for source code management. It is a free, open-source distributed version control system that tracks changes in any set of files, enabling multiple developers to work together on a project without overwriting each other's contributions.

Created by Linus Torvalds in 2005 for managing the development of the Linux kernel, Git has become the standard version control system used across the software industry.

Key Concepts

  • Repository (repo) — A project folder tracked by Git, containing all files and their complete history of changes.
  • Commit — A snapshot of your changes at a point in time, with a descriptive message explaining what was changed and why.
  • Branch — An independent line of development that lets you work on features or fixes without affecting the main codebase.
  • Merge — The process of integrating changes from one branch into another.

Why Git Matters for DevOps

In a DevOps world, everything is version-controlled — not just application code, but also infrastructure definitions (IaC), CI/CD pipeline configurations, documentation, and operational scripts. Git is the single source of truth that makes collaboration, automation, and auditability possible.

What is GitHub?

GitHub is an online software development platform used for storing, tracking, and collaborating on software projects. It hosts Git repositories in the cloud and layers on powerful collaboration features such as pull requests, issues, project boards, and GitHub Actions for CI/CD.

Git vs GitHub

Git is the version control tool that runs on your local machine. GitHub is a hosting platform that adds collaboration features on top of Git. You can use Git without GitHub, but GitHub needs Git to function.

Other Platforms

  • GitLab — An all-in-one DevOps platform with built-in CI/CD, container registry, and more.
  • Bitbucket — Atlassian’s Git hosting service, integrates tightly with Jira and Trello.
  • Azure DevOps — Microsoft’s platform combining repos, pipelines, boards, and artifacts.

SSH Keys

SSH keys allow you to authenticate with GitHub (or GitLab) securely without entering your username and password each time you push or pull.

How to Add an SSH Key to GitHub

1

Open your terminal

2

Run the following command to display your public key:

bash
cat ~/.ssh/id_rsa.pub
3

Copy the entire contents of the output to your clipboard

4

Log in to your GitHub account

5

Click on the drop-down menu (your profile icon in the top-right) → Settings

6

Navigate to "SSH and GPG keys" in the left sidebar

7

Click "New SSH key"

8

Enter a descriptive name in the Title field (e.g., "Work Laptop")

9

Paste your key in the Key field

10

Click "Add SSH key"

How to Generate an SSH Key

If you don't have an SSH key yet, generate one with the following command:

bash
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"

Follow the prompts to choose a file location (press Enter for the default) and optionally set a passphrase. Your key pair will be saved to ~/.ssh/.

How to Delete an SSH Key

To remove an SSH key from your GitHub account, go to Settings → SSH and GPG keys and click "Delete" next to the key you want to remove.

Personal Access Tokens

Personal access tokens (PATs) are an alternative to SSH keys for authenticating with GitHub, especially useful for HTTPS-based operations and API access.

1

Log in to your GitHub account

2

Click on the drop-down menu (profile icon) → Settings

3

Scroll down to "Developer settings" at the bottom of the left sidebar

4

Click "Personal access tokens"

5

Choose between "Fine-grained tokens" (recommended) or "Tokens (classic)"

6

Click "Generate new token"

7

You may be asked to confirm access — enter your password if prompted

8

Configure a token name, expiration date, and select the required permissions / scopes

9

Copy and save the token immediately — you won’t be able to see it again!

Git Fundamentals

These are the core commands you will use every day. Understanding them is essential before moving on to branching strategies and collaboration workflows.

bash
# Initialize a new repository
git init

# Clone an existing repository
git clone <url>

# Check the status of your working directory
git status

# Stage changes for commit
git add <file>        # Stage a specific file
git add .             # Stage all changes

# Commit staged changes
git commit -m "Your commit message"

# Push changes to remote
git push origin <branch>

# Pull latest changes from remote
git pull origin <branch>

Branching & Merging

A branch is an independent line of development. Branches let you work on features, bug fixes, or experiments in isolation without affecting the main codebase.

Branch Commands

bash
# Create and switch to a new branch
git checkout -b feature/my-feature

# List all branches
git branch -a

# Switch branches
git checkout main

# Merge a branch into current branch
git merge feature/my-feature

# Delete a branch
git branch -d feature/my-feature

Branching Strategies

Gitflow

A structured model with dedicated branches for features, releases, and hotfixes. Best for projects with scheduled releases.

main develop feature/* release/* hotfix/*

Trunk-Based Development

Short-lived feature branches merged frequently into main. Encourages continuous integration and reduces merge conflicts.

main short-lived branches

GitHub Flow

A simple, lightweight model: main plus feature branches with pull requests. Ideal for continuous deployment.

main feature branches pull requests

Rebasing vs Merging

Merge preserves the full history by creating a merge commit that ties two branches together. It shows exactly when branches diverged and converged.

Rebase rewrites the commit history by replaying your changes on top of the target branch, producing a clean, linear log. While this makes the history easier to read, it should never be used on shared/public branches since it rewrites commits other people may depend on.

Pull Requests & Code Review

A pull request (PR) is a proposal to merge changes from one branch into another. It is the primary mechanism for code review and collaboration in modern software development.

PR Workflow

1

Create a new branch from main

2

Make your changes and commit them

3

Push the branch to the remote repository

4

Open a pull request on GitHub (or your platform of choice)

5

Team members review the code, leave comments, and suggest changes

6

Address feedback, push additional commits if needed

7

Once approved, merge the pull request into main

Code Review Best Practices

  • Review for logic, not style — Use linters and formatters to handle style; focus your review on correctness, security, and maintainability.
  • Be constructive — Suggest improvements rather than simply pointing out problems. Explain why something should change.
  • Approve or request changes — Don’t leave a review in limbo. Either approve, request specific changes, or leave a comment.

Best Practices

Follow these guidelines to keep your repositories clean, your team productive, and your secrets safe.

  • ✓
    Use .gitignore — Never commit secrets, credentials, node_modules, build artifacts, or OS-specific files. Create a .gitignore at the root of every repo.
  • ✓
    Write meaningful commit messages — Describe what changed and why, not just what files were touched. Good messages make debugging and auditing much easier.
  • ✓
    Keep commits small and focused — Each commit should represent one logical change. This makes reviews simpler and reverts safer.
  • ✓
    Use branch protection rules — Require code reviews and passing CI checks before merging to main. This prevents accidental breakage.
  • ✓
    Never commit secrets — Use environment variables, secret managers (e.g., HashiCorp Vault, AWS Secrets Manager), or .env files that are git-ignored.
  • ✓
    Pull before push — Always run git pull before git push to incorporate remote changes and avoid unnecessary merge conflicts.
  • ✓
    Tag releases — Use semantic versioning (v1.0.0, v1.1.0, etc.) to mark release points. Tags provide clear reference points in your project history.
